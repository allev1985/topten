# myfaves Constitution

## Overview

**myfaves** (`topten`) is a Next.js App Router application where users curate
and share lists of favourite places. Creators build personalised, visually-focused
lists; visitors discover and browse them. The platform is deployed on Vercel with
Supabase for auth and Postgres, Drizzle ORM, Tailwind CSS v4, shadcn/ui, and the
Google Places API for place data.

This constitution governs all development decisions. It supersedes informal
preferences and individual choices whenever they conflict.

---

## Core Principles

### I. Code Quality & Maintainability (NON-NEGOTIABLE)

Code MUST be simple, readable, and maintainable by any contributor familiar with
the stack. Every module MUST have a clear, single responsibility. Avoid
unnecessary abstractions — complexity MUST be explicitly justified in
`docs/decisions/` or inline comments linked to the relevant ADR.

**DRY**: Duplicated logic MUST be extracted into reusable modules, utilities,
or shared components. Before implementing any functionality, contributors MUST
verify that similar logic does not already exist. Unavoidable duplication MUST
be documented.

**Framework Code Integrity**: Code generated by frameworks or CLI tools —
specifically shadcn/ui components in `src/components/ui/` — MUST NOT be
modified directly. Customise by composition (wrapping) or by shadcn/ui's own
configuration and variant system. Direct modifications are only permitted when
no other option exists and MUST be recorded in a decision record.

**Rationale**: High quality and DRY reduce defects, lower maintenance burden,
and make iterative change safe and predictable.

### II. Testing Discipline & Safety Nets (NON-NEGOTIABLE)

All user-facing behaviour and critical logic MUST be covered by automated tests.

- Tests MUST be written or updated before or alongside implementation.
- The full test suite MUST pass before merge.
- Bug fixes MUST include a failing test that reproduces the issue before the fix is applied.
- Critical paths MUST have unit, integration, and — where applicable — E2E tests.

**Test layers** (Vitest unless otherwise noted):
- `tests/unit/` — business logic, utilities, hooks
- `tests/component/` — React Testing Library, UI in isolation
- `tests/integration/` — auth flows, middleware, service-to-Supabase
- `tests/e2e/` — Playwright, critical user journeys

**Rationale**: Strong tests provide confidence to refactor, prevent regressions,
and document expected behaviour.

### III. User Experience Consistency

User-facing flows MUST feel consistent across the application. Terminology,
interaction patterns, and visual structure SHOULD match across similar pages.
Breaking changes to UX (navigation, route structure, form behaviour) MUST be
deliberate, communicated, and — when possible — backward-compatible or phased.

**Rationale**: Consistency reduces confusion and builds user trust.

### IV. Performance & Resource Efficiency

Features MUST be designed to meet clearly defined performance targets appropriate
to their scope (latency, throughput, memory usage). Baseline expectations MUST be
captured in plan or spec documents before heavy implementation begins. Changes
that materially impact performance MUST be measured, and optimisations MUST
preserve correctness and readability.

**Rationale**: Predictable performance ensures myfaves remains reliable and
responsive as usage grows.

### V. Observability & Debuggability

The system MUST provide enough logging and diagnostics to understand behaviour in
both development and production-like environments. Logs MUST be structured where
feasible and MUST NOT leak sensitive data (tokens, passwords, PII). Errors MUST
surface actionable messages for both users and developers.

**Rationale**: Good observability shortens debugging cycles and makes reliability
work practical.

### VI. Architecture Integrity (NON-NEGOTIABLE)

myfaves uses a **service-based architecture**. Every domain has a dedicated
service module that owns all business logic and database access for that domain.

#### Server Action layer contract

Server Actions (`src/actions/`) are **thin coordination layers only**. Their
sole responsibilities are:

1. **Authenticate** — verify the session via `getSession()`; return an auth
   error `ActionState` immediately if the user is not authenticated.
2. **Validate input** — run Zod schema validation with `safeParse()`; return
   field-level `ActionState` errors immediately on failure.
3. **Delegate to the service** — call the appropriate domain service function
   (e.g., `updateSlug()` from `src/lib/profile/service.ts`).
4. **Map service errors** — catch `ProfileServiceError`, `AuthServiceError`, or
   equivalent typed errors and convert them to the correct `ActionState` shape.
5. **Revalidate** — call `revalidatePath()` on success and return success
   `ActionState`.

**Server Actions MUST NOT contain**:
- Direct Drizzle / database calls (no `db.select`, `db.update`, etc.)
- Raw SQL or query logic
- Business rules, uniqueness logic, or complex conditional flows

Anything beyond the five steps above belongs in the service layer.

#### Service layer contract

Domain services (`src/lib/{domain}/service.ts`) are the **single source of
truth** for business logic and database access within their domain.

- The auth service (`src/lib/auth/service.ts`) owns all auth operations.
- The profile service (`src/lib/profile/service.ts`) owns all profile mutations.
- Each service MUST expose typed result objects and typed error classes
  (`src/lib/{domain}/service/types.ts`, `src/lib/{domain}/service/errors.ts`).
- Services MUST NOT call other services directly — co-ordination belongs in the
  action layer or a dedicated orchestration module if complexity warrants it.
- External integrations (Google Places) MUST flow through their own service
  module and MUST be called server-side only.

#### API routes

New API Routes MUST NOT be created unless there is an explicit callback
requirement (e.g., the existing `GET /api/auth/verify`). All mutations go
through Server Actions.

**Rationale**: A strict two-layer contract (action = coordination, service =
logic) keeps business rules testable in isolation, prevents scattered database
code, eliminates duplication, and makes each layer independently verifiable.

### VII. Data Integrity

- **Soft deletes everywhere**: Records MUST be soft-deleted by setting
  `deleted_at`. Hard deletes are forbidden for MVP data (users, lists, places,
  list-places). This preserves referential integrity and enables recovery.
- **Every query filtering user data MUST include `deleted_at IS NULL`**. Omitting
  this filter is a defect.
- List slugs are unique per user, not globally. Queries against slugs MUST scope
  to the owning user.
- All primary keys are UUIDs.

**Rationale**: Soft deletes reduce risk of irreversible data loss and maintain
consistent referential integrity across related tables.

### VIII. Security Boundaries

- The Google Places API key (`GOOGLE_PLACES_API_KEY`) is server-side only. It
  MUST NOT be prefixed `NEXT_PUBLIC_` and MUST NOT appear in any client bundle.
- `SUPABASE_SERVICE_ROLE_KEY` MUST only be used in trusted server-side contexts
  (seed scripts, admin operations). It MUST NOT be exposed to the client.
- Redirect targets after login/logout MUST be validated to prevent open-redirect
  attacks. Use the existing `redirectTo` validation helper.
- Password requirements: 12+ characters, containing uppercase, lowercase,
  number, and special character. This constraint MUST NOT be relaxed without a
  decision record.

**Rationale**: A clear security boundary between client and server prevents
credential leakage and common web vulnerabilities.

## Technology Stack & Constraints

| Layer | Technology | Key Rule |
|-------|-----------|----------|
| Framework | Next.js App Router (Vercel) | Prefer Server Components; use Client Components only for interactivity |
| Auth & DB | Supabase (Postgres + Auth) | All auth logic via `src/lib/auth/service.ts` |
| ORM | Drizzle | Schema in `src/db/schema/`; run migrations before deploying schema changes; direct DB calls in service layer only |
| Profile | Profile service (`src/lib/profile/service.ts`) | All profile mutations via this service; never called directly from components |
| UI | Tailwind CSS v4 + shadcn/ui | Never edit `src/components/ui/` directly |
| Places | Google Places API | Server-side only; cache results in `Place` table |
| Testing | Vitest + Playwright | Tests required before merge |
| Package manager | pnpm | Do not mix with npm or yarn |

## URL & Routing Conventions

| Route pattern | Description |
|---------------|-------------|
| `/@{vanity_slug}` | Creator profile |
| `/@{vanity_slug}/{list-slug}` | Individual list |
| `/dashboard` | Authenticated creator workspace |
| `/dashboard/settings` | Account settings |

- `/dashboard` and `/dashboard/settings` are protected by middleware.
- Public routes: `/`, `/login`, `/signup`, `/verify-email`, `/forgot-password`,
  `/reset-password`.
- New routes MUST be registered in `src/lib/config/` (route constants) and
  middleware must be updated if protection is required.

## Quality & Delivery Standards

- Every feature plan MUST document: testing strategy, performance goals, and any
  deliberate UX changes.
- Specifications MUST define testable acceptance criteria for each user story,
  including clear success and failure states.
- Tasks MUST be traceable to the user stories and principles they enforce.

---

## Delivery Workflow & Review Gates

No code MAY be merged to `main` without:

1. Passing tests for all impacted areas.
2. A code review that explicitly checks against the Core Principles.
3. Updated documentation where behaviour, UX, routing, or data contracts change.

Risky changes — schema migrations, public-facing URL changes, auth flow changes,
major UX shifts — MUST reference a decision record in `docs/decisions/`.

---

## Governance

The myfaves Constitution supersedes informal practices and individual preferences
when they conflict. All contributors are responsible for understanding and
applying these principles.

- **Amendments**: Any proposal to change principles or governance MUST be
  documented, reviewed, and approved via standard code review. The Sync Impact
  Report at the top of this file MUST be updated with the version bump, affected
  sections, and any follow-up TODOs.
- **Versioning** follows semantic versioning:
  - **MAJOR**: Breaking changes to principles or governance (e.g., removing a
    NON-NEGOTIABLE constraint, changing the architecture pattern).
  - **MINOR**: New principles or material expansions.
  - **PATCH**: Clarifications that do not change expected behaviour.
- **Compliance Review**: Feature plans, specs, and task lists MUST include an
  explicit "Constitution Check" step verifying alignment with code quality,
  testing, architecture, data integrity, and security requirements.
